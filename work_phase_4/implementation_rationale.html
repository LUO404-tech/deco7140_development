Implementation Rationale

1. Alignment with Client and User Needs

In WP3 I defined the audience as people who enjoy hobby and subculture communities (cosplay, anime, dance, crafting, indie music, and related interests). When building WP4 I kept the same audience, so the website stays interest-driven, ESL-friendly, and easy to browse.

Information architecture reused from WP3: I kept the WP3 site map – Home, Introduction, News, Community Hubs, Learning Studio, Community Support, Login – so anyone who saw the WP3 prototype can recognise the structure and new visitors can navigate by page purpose.

Homepage welcome text reused: the welcome on homepage.html keeps the inclusive, community-first tone I wrote in WP3, so WP4 pages still speak to “people who like these communities”, not to a narrow campus-only group.

Learning Studio added for newcomers: I added a Learning Studio page with Beginner Guides and Video Tutorials so people who are new to a hobby can get context before visiting Community Hubs or posting.

WP3 API practice carried over: WP4 uses the same pattern from WP3: one GET to load current community voices, and one POST (“Join the Community”) so users can contribute without a complex sign-up.

Form style kept in the same system: the login page follows the WP3 form style (bold labels, rounded white card, blue primary action) so the new page belongs visually to the same project, even though WP4 added a second Clear button.

Card components inherited: I reused the rounded, shadowed cards from WP3 (Values Mapping Table, reflective pages) for the hubs hero cards and the join form. This makes WP4 visually consistent with the WP3 deliverables.

Connection to CCI: content is grouped by interests, input fields are low-barrier, and labels are clear – this aligns with the Connect Communities Initiative goal of helping people join or create communities quickly.

2. Accessibility and Ethical Considerations

I did not restart accessibility work from zero. Instead, I applied the WP2/WP3 principles to the new API-based pages and to the login flow.

Keyboard and focus support: filter chips on hubs.html are buttons with aria-pressed and can be triggered with Enter/Space so keyboard-only users can change the active community.

ESL-friendly copy: forms use short, typical labels (“Email”, “Password”, “Message (optional)”), based on the WP3 style guide and reflective design page.

Minimal data collection: the community POST form only sends what the DECO7140 API needs (name, email, category, message, optional image). This continues the privacy decisions from the WP3 Values Mapping Table.

Learning as inclusion: providing beginner material in the Learning Studio prevents new users from being excluded just because they do not know the subculture terms yet.

Inclusive language carried over: WP4 keeps the “everyone / users / people” wording from WP3 so content stays neutral and respectful.

3. Variations from A3 Design Proposal

When I connected the real API and tested the pages, I made a few adjustments. All of them stay inside the original concept.

Inline reply on community posts: because the API does not provide threaded replies but A3 emphasised interaction, I added a local inline-reply UI under each card so users can respond in context without breaking the endpoint.

Hashtag prefixes for filtering: posts are prefixed with tags like #cosplay or #anime so the client-side filter can reliably group voices by interest.

Login actions extended: instead of a single “Submit” button like the WP3 demo, the WP4 login card has both Sign in and Clear so users can correct mistakes quickly, especially on mobile.

Refreshed blue palette: I used the WP4 header blue on cards and buttons, instead of copying the WP3 demo colour exactly. This keeps contrast good enough for the audit and matches the final layout.

Image sizing adjusted: some hero/card images are tighter than the A3 mockups to load faster on UQCloud.

Login added to global nav: I exposed login.html in the shared navigation so the marker can reach the form from any page.

4. Challenges and Strategies

Flat API → multiple views: the endpoint returns one flat list, but I needed “by community” views. I solved it on the client side with filtering and status messages so users can switch categories without reloading.

Keeping navigation consistent: all WP4 pages reuse the same header markup and JS toggle. After I added the Login item, I only updated this shared piece once.

Form UX for mixed-ability users: I kept inputs short, marked optional fields clearly, and added success/failure feedback after POST so users always know whether the API call worked.

Visual continuity with WP3: I matched WP3 typography, spacing, and card borders so WP4 still “looks like” the project shown in the Behavioural / Reflective design pages.

5. Accessibility Audit

I ran the IBM Equal Access Accessibility Checker (Chrome DevTools) on work_phase_4/hubs.html on 31 Oct 2025. I picked this page because it is the most complete WP4 example: shared navigation, hero cards, filter chips, API-loaded voices, and the “Join the Community” POST form.

Environment: Chrome 141, DevTools device preview iPhone SE (375px), http://127.0.0.1:5501/work_phase_4/hubs.html.

Main violation – SVG accessible names: the scan reported several “The SVG element has no accessible name” issues. These come from the decorative hero illustrations inherited from WP3. I kept the icons for visual support and documented the issue.

Focus indicator advisory: the tool reminded me to check keyboard focus indicators for the filter chips because they use custom borders. I kept the current look so the page still matches the WP3 style guide and noted it as a future improvement.

Colour / blue scheme: the checker warned to verify that colour is not the only cue. The chips already use aria-pressed, so I kept the WP4 blue scheme.

Because these SVGs are non-interactive and came from the WP3 visual language, I chose to keep them and record the finding here instead of removing the images.
